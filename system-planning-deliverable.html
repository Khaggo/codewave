Business Scope and Platform Framing

The updated paper frames the system as a service management and customer engagement platform centered on vehicle-based lifecycle tracking. The primary value lies in booking management, service and insurance monitoring, customer interaction, product ordering, inventory visibility, and decision support through a centralized platform.

Main Service Ownership E-Commerce Ownership Async / Integration Layer

Primary Objective

Streamline service operations, enhance customer interaction, and improve business efficiency through centralized booking, product transactions, insurance inquiries, and vehicle-based record monitoring.

Core System Lens

Use a vehicle-centric model so service history, bookings, and insurance-related transactions appear in a unified timeline for both administrators and customers.

Critical Principle

Use explicit service ownership. The main service may aggregate or proxy e-commerce data to the frontend, but products, inventory, invoice orders, and payment tracking state remain under the e-commerce backend.

Core Assumptions

These assumptions shape the proposed architecture and data model.

Inspection-Backed Accuracy

Vehicle lifecycle entries may be system-generated for transactional events, but condition-sensitive milestones should be backed by staff inspection records so the timeline remains credible.

Job Monitoring Scope

The system monitors service progress in line with the existing job order and job board workflow. It does not replace the client’s established operational process with autonomous job-order generation.

Insurance Boundaries

The insurance module supports quotation inquiry intake and tracking. It does not directly integrate with insurance providers, generate policies, or automate approvals.

Payment Boundaries

E-commerce payment handling is invoice-based and tracking-focused. The system monitors statuses such as pending, partial, and paid, without automated verification or bank integration.

Platform Split

The system remains a web-based administrative platform plus a customer-facing mobile application. Operational analytics come from the main domain, while commerce summaries arrive through service APIs or events.

End-to-End System Flow

The flow below now follows the updated paper plus the approved update for inspection-backed lifecycle accuracy and back-job handling.

1. Account Creation and Customer Profile Setup

User registers, logs in, fills profile details, and adds one or more vehicles with plate number, type, and model. These become the base context for service history, booking records, insurance activity, and personalized vehicle lifecycle monitoring.

2. Service Discovery and Booking Creation

Customer browses available auto-care services such as car wash, detailing, and PMS, selects the target vehicle, chooses real-time or scheduled booking, and reserves a time slot.

3. Booking Validation and Confirmation

Main service validates slot availability, records the booking, stores selected services, and issues confirmation. A booking status history entry is written for traceability, while the vehicle lifecycle timeline is updated with the new administrative milestone.

4. Intake Inspection and Job Monitoring

When the vehicle arrives, staff can record an intake inspection to verify the vehicle's actual condition, complaint context, and service scope. Admin or staff then monitors service progress in alignment with the existing job order and job board workflow.

5. Reminder and Notification Triggering

Once a booking is confirmed, background jobs schedule email and SMS reminders. Separate rules also generate insurance renewal alerts and service reminders based on vehicle history and previous transactions.

6. Completion Inspection and Loyalty Accrual

Before the work is treated as fully verified, staff can record a completion inspection. After that, the booking status is updated, customer activity is logged, points are credited to the loyalty account, and the vehicle timeline is extended with a verified completion event.

7. Insurance Inquiry Intake

Customer can submit insurance quotation requests with optional document uploads. The inquiry moves through tracking statuses while uploaded files are linked and auditable, without requiring direct insurer integration.

8. Vehicle Lifecycle Tracking Access

Both customer and admin can view a consolidated vehicle timeline that combines service history, booking records, insurance-related activities, and inspection-backed condition milestones in one organized record-monitoring view.

9. Back Job Intake and Return Inspection

If a customer returns after a repair or service, staff can open a back job linked to the original work, inspect the returning issue, determine whether it is related rework or a new concern, and track resolution status through the same operational workflow.

10. Chatbot-Guided Inquiry Handling

The customer-facing side includes a rule-based chatbot that responds to common service, booking, insurance, back-job, and product questions using guided flows rather than AI-generated answers.

11. E-Commerce Order Journey

Customer browses products, adds items to cart, generates an invoice-based order, tracks the order, and later reviews purchase history. The e-commerce service owns products, carts, orders, payment-status tracking, and inventory data.

12. Inventory and Payment Tracking Updates

Order creation updates inventory reservations and creates payment-tracking records with invoice-oriented states such as pending, partial, and paid. Manual verification and manager-side confirmation may still exist outside full automation.

13. Admin Dashboard and Reporting

Dashboard views combine booking volumes, customer activity, service trends, inquiry workload, inspection turnaround, back-job counts, repeat issues, and commerce summaries. Aggregation jobs can run asynchronously to keep read performance stable.

A two-domain microservice direction still fits the updated paper because operations/lifecycle management and e-commerce/inventory/payment tracking have different ownership, scaling patterns, and data-change rhythms.

Recommended Communication Pattern

Use the simplest transport that fits the responsibility.

Synchronous HTTP

Use for login, profile reads, slot queries, booking creation, catalog browsing, cart changes, and checkout requests that need immediate client responses.

BullMQ + Redis

Use for delayed or retryable internal jobs: reminders, status follow-ups, vehicle-timeline refresh, analytics rebuilds, chatbot support jobs, and inventory alert processing.

RabbitMQ

Use for cross-domain events where one service publishes business facts and another reacts without tight runtime coupling.

What You Should Account For

These are the implementation and design concerns most likely to cause gaps if left implicit under the updated paper.

Status Lifecycles Must Be Explicit

  • Bookings need lifecycle states such as draft, pending, confirmed, rescheduled, in-progress, completed, declined, cancelled, no-show.
  • Insurance inquiries need pending, processing, approved, rejected, closed, or follow-up-needed states.
  • Orders need their own commerce lifecycle and should not reuse booking states.

History and Auditability Matter

  • Write history tables for booking status changes and possibly inquiry status changes.
  • Use audit logs for admin actions, reward adjustments, manual overrides, and manager approval decisions.
  • Do not overwrite important workflow states without preserving a trace.

Uploads Need Metadata and Security

  • Store upload metadata in file_uploads and reference it from insurance documents, vehicle records, or job-monitoring attachments.
  • Validate file type, file size, access ownership, and retention policy.
  • Keep actual file storage decoupled from relational records.

Lifecycle Tracking Needs a Clear Model

  • The paper’s strongest differentiator is the vehicle-centric timeline that merges bookings, services, and insurance activities.
  • Implement it with lifecycle events plus inspection records so condition-based milestones can be marked as verified only after physical inspection.
  • Background queueing via BullMQ is infrastructure, not the same thing as lifecycle/job monitoring.

Chatbot Must Stay Rule-Based

  • The paper specifies a guided-response inquiry bot, not AI automation.
  • Model intents, fallback paths, canned answers, and escalation to manual contact when the flow cannot resolve the question.

Back Job Needs Explicit Linkage

  • Treat a back job as a return or rework case tied to an original completed job or service record, not as an ordinary new visit.
  • Capture complaint, relation to prior work, return inspection findings, resolution status, and closure notes.

Job Monitoring Should Respect Existing Workflow

  • Track progress, milestones, assignees, and status notes without forcing a brand-new autonomous job-generation flow.
  • Make the system compatible with the current job order and job board process already used by the client.

Loyalty Across Two Domains Needs Contract Discipline

  • Do not let the e-commerce service write directly into main-service loyalty tables.
  • Publish a commerce event with customer reference, order amount, and rewardable metadata.
  • Main service calculates and writes the points transaction.

Inventory and Invoice Tracking Need Operational Discipline

  • Inventory applies to automotive products and service-related materials, so the schema should support multiple stock contexts if needed.
  • Invoice payment tracking should support pending, partial, and paid states, due dates, reference numbers, and manual validation notes.

BullMQ in NestJS

Reminder Queue

Good fit for booking reminders, insurance renewals, and notification retries.

@Module({
  imports: [
    BullModule.forRoot({
      connection: { host: process.env.REDIS_HOST, port: 6379 },
    }),
    BullModule.registerQueue({ name: 'notifications' }),
  ],
})
export class QueueModule {}

@Injectable()
export class ReminderService {
  constructor(@InjectQueue('notifications') private queue: Queue) {}

  scheduleBookingReminder(data: ReminderPayload) {
    return this.queue.add('booking-reminder', data, {
      delay: data.delayMs,
      attempts: 3,
      backoff: { type: 'exponential', delay: 1000 },
    });
  }
}
BullMQ Worker

Processor / Worker Host

Use dedicated workers for delivery and analytics jobs to keep the API responsive.

@Processor('notifications')
export class NotificationProcessor extends WorkerHost {
  async process(job: Job<NotificationJob>) {
    switch (job.name) {
      case 'booking-reminder':
        return this.notificationGateway.send(job.data);
      case 'insurance-renewal':
        return this.notificationGateway.send(job.data);
    }
  }

  @OnWorkerEvent('failed')
  onFailed(job: Job, error: Error) {
    this.logger.error(`Job ${job.id} failed: ${error.message}`);
  }
}
Drizzle ORM + PostgreSQL

Main Service Schema Style

Typed relational schema works well for bookings, vehicles, and history tables.

export const bookings = pgTable('bookings', {
  id: serial('id').primaryKey(),
  userId: integer('user_id').notNull().references(() => users.id),
  vehicleId: integer('vehicle_id').notNull().references(() => vehicles.id),
  timeSlotId: integer('time_slot_id').references(() => timeSlots.id),
  status: varchar('status', { length: 40 }).notNull(),
  scheduledAt: timestamp('scheduled_at', { withTimezone: true }).notNull(),
  createdAt: timestamp('created_at', { withTimezone: true }).defaultNow().notNull(),
});

export const bookingStatusHistory = pgTable('booking_status_history', {
  id: serial('id').primaryKey(),
  bookingId: integer('booking_id').notNull().references(() => bookings.id),
  status: varchar('status', { length: 40 }).notNull(),
  changedBy: integer('changed_by').references(() => users.id),
  changedAt: timestamp('changed_at', { withTimezone: true }).defaultNow().notNull(),
});
